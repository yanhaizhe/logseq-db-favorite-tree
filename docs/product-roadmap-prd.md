# 产品路线图 PRD

本文档用于定义 `Logseq DB Favorite Tree` 下一阶段的产品路线图、优先级判断依据、范围边界与验收方向，作为排期、实现与评审的统一基线。

## 1. 文档目标

- 为后续迭代提供明确的优先级顺序，而不是继续堆叠零散想法
- 保持插件定位稳定：以“基于收藏与属性关系的轻量导航器”为核心
- 将高价值、低风险、符合边界的需求优先落地
- 明确哪些能力暂不进入路线图，避免范围膨胀

## 2. 当前产品定位

- 根节点来自 Logseq 收藏页
- 层级关系来自页面属性，默认 `parent`
- 支持 `sidebar`、`floating`、`mixed` 三种显示模式
- 核心能力是浏览、搜索、定位、排序、布局与状态恢复
- 当前产品优先解决“导航效率”，而不是承担“页面结构编辑器”职责

## 3. 路线图判断原则

### 3.1 优先级标准

- **价值密度**：是否直接提升高频导航、定位与整理效率
- **边界匹配**：是否仍然符合“轻量导航器”定位
- **实现风险**：是否会显著抬高状态复杂度、误操作风险或维护成本
- **可验证性**：是否容易定义验收标准并在真实使用中验证效果

### 3.2 本阶段策略

- 先补强已有能力的可治理性与完成度
- 再增加少量低风险写操作，提升日常使用频率
- 最后再考虑复杂场景能力与性能补强
- 暂不进入会改变产品性质的重型需求

## 4. 核心用户与场景

### 4.1 目标用户

- 在 Logseq DB graph 中使用收藏夹组织工作流的用户
- 依赖页面属性维护父子关系的重度笔记用户
- 需要在“当前页上下文”和“全局结构导航”之间快速切换的用户

### 4.2 关键场景

- 在侧边栏或悬浮面板中快速找到当前项目上下文
- 在搜索结果中定位某条路径并跳转到目标页面
- 在不打断写作流的前提下，快速整理收藏根和同级顺序
- 在多个 graph 间保持各自独立的导航布局与状态

## 5. 当前问题总结

- 已有排序能力，但缺少恢复、清理与状态标识，治理能力不足
- 搜索与定位已可用，但命中切换和路径聚焦还不够高效
- 新用户在“没有收藏”或“没有层级关系”时缺少足够明确的引导
- 当前操作几乎都偏只读，缺少少量高频低风险的快捷动作
- 大树场景虽未必已成为普遍瓶颈，但应提前纳入性能观察范围

## 6. 路线图总览

### 6.1 P0：高价值低风险，优先落地

#### P0-1 排序治理能力补齐

- 增加“默认排序 / 自定义排序”切换入口
- 切换到默认排序后，保留当前层已保存的自定义排序结果，支持随时切回
- 增加“清除当前层自定义排序”独立操作
- 为存在自定义排序的层级增加轻量视觉状态标记

**价值**

- 补齐当前排序能力的闭环
- 降低用户因拖拽后无法回退带来的心理负担
- 强化“默认排序 + 自定义覆盖 + 可回切”的产品心智

**验收方向**

- 用户可在根层或任意已排序层切换查看默认排序结果
- 切换到默认排序后，用户仍可一键切回此前保存的自定义排序
- 清除当前层自定义排序前，系统提供二次确认，避免误清除
- 清除当前层排序后，已保存的自定义顺序被真正移除，新旧节点回到标题排序逻辑
- 用户能明确识别当前层是否已存在自定义排序

#### P0-2 拖拽反馈优化

- 增加更明显的落点提示
- 增加拖拽过程中的可见反馈态
- 明确显示“不可投放”状态，减少误解

**价值**

- 直接提升现有核心交互的可理解性
- 减少拖拽排序中的犹豫与误操作

**验收方向**

- 用户能清楚看到投放前后位置
- 无效拖拽不会造成“像成功又像失败”的模糊体验

#### P0-3 空状态与异常提示优化

- 在无收藏时提供明确操作引导
- 在仅有根节点但未形成层级关系时给出轻量提示
- 对刷新失败、解析失败、无命中等状态提供更可操作的说明

**价值**

- 提升首次使用成功率
- 减少“插件没生效”的误判

**验收方向**

- 新用户打开插件后能理解下一步该做什么
- 常见异常状态有明确、简洁、可执行的提示

#### P0-4 在右侧边栏打开

- 在节点操作中提供“在右侧边栏打开”入口

**价值**

- 贴合 Logseq 的高频工作流
- 在不改产品边界的前提下补齐导航场景

**验收方向**

- 用户可从树节点直接把页面打开到右侧边栏
- 不影响原有“在主区域打开页面”的默认行为

### 6.2 P1：提升日常使用频率

#### P1-1 从当前页一键加入收藏

- 提供从当前页面加入收藏的快捷入口

**价值**

- 提升冷启动效率
- 降低用户从“想到收藏”到“完成收藏”的操作成本

**验收方向**

- 用户在当前页面可快速将其加入收藏根
- 加入后能在树中及时看到结果

#### P1-2 只展开当前路径 / 折叠其他分支

- 提供“只保留当前路径展开”的专注式浏览
- 提供“折叠其他分支”的快速收敛能力

**价值**

- 适合大树场景下快速收束上下文
- 与定位当前页、搜索结果浏览形成自然协同

**验收方向**

- 用户可在一次操作后快速聚焦当前上下文
- 不破坏已有展开状态的整体可预期性

#### P1-3 搜索多命中切换导航

- 提供上一条 / 下一条命中跳转
- 自动滚动到当前命中项并维持可见反馈

**价值**

- 提升搜索结果浏览效率
- 比单纯高亮路径更直接解决“如何快速跳命中”的问题

**验收方向**

- 搜索到多个命中时，用户可顺序切换目标
- 当前命中有明显定位反馈

#### P1-4 全局快捷键呼出面板

- 提供快捷键呼出悬浮面板
- 可选自动聚焦到搜索框

**价值**

- 降低从写作流切入导航的阻力
- 提升重度用户的操作效率

**验收方向**

- 快捷键触发稳定且不与现有主流程冲突
- 呼出后可直接继续搜索或浏览

### 6.3 P2：增强复杂场景体验

#### P2-1 节点轻量操作菜单

- 为节点补充少量高频操作入口
- 控制在 2 到 3 个动作以内，避免菜单膨胀

**建议候选动作**

- 在右侧边栏打开
- 移出收藏
- 定位到当前节点路径或聚焦该分支

#### P2-2 快捷新建子节点

- 在树节点上快捷创建子页面
- 自动写入目标父页面属性关系

**前置条件**

- 需要先定义页面重名、写入失败、回滚与提示策略
- 需要明确该功能是“增强输入效率”，不是“替代 Logseq 页面编辑器”

#### P2-3 图标 / Emoji 展示

- 在页面具备可识别图标信息时进行展示

**价值**

- 增强扫描效率
- 提升树的可辨识度

**优先级说明**

- 感知强，但不解决当前最核心的问题，因此后置

#### P2-4 大树性能基线与渐进优化

- 建立大树场景下的性能观察基线
- 优先做轻量优化，再判断是否需要进入重型虚拟化方案

**关注点**

- 大量节点展开时的渲染成本
- 搜索与刷新同时发生时的交互流畅度
- 定位与状态恢复在深层树中的稳定性

## 7. 暂不进入路线图

### 7.1 跨层级拖拽改写父子关系

- 暂不支持通过拖拽直接改写 `parent` 属性

**原因**

- 与当前产品边界冲突
- 容易引发误操作与数据一致性问题
- 会把插件从“导航器”推向“结构编辑器”

### 7.2 多属性 / 多维聚合视图

- 暂不提供按多个属性切换树视图

**原因**

- 会显著扩大产品定位
- 需要额外的视图模型、切换逻辑与复杂认知成本

### 7.3 颜色标签系统

- 暂不引入基于属性的颜色编码

**原因**

- 规则不稳定时容易制造视觉噪音
- 当前信息密度已足够，不急于叠加更多视觉通道

### 7.4 重型虚拟列表改造

- 暂不直接进入虚拟滚动或大规模渲染架构改造

**原因**

- 当前应先建立真实性能瓶颈证据
- 过早架构升级会显著提高维护成本

## 8. 建议排期顺序

### 8.1 第一阶段

- P0-1 排序治理能力补齐
- P0-2 拖拽反馈优化
- P0-3 空状态与异常提示优化
- P0-4 在右侧边栏打开

### 8.2 第二阶段

- P1-1 从当前页一键加入收藏
- P1-2 只展开当前路径 / 折叠其他分支
- P1-3 搜索多命中切换导航
- P1-4 全局快捷键呼出面板

### 8.3 第三阶段

- P2-1 节点轻量操作菜单
- P2-2 快捷新建子节点
- P2-3 图标 / Emoji 展示
- P2-4 大树性能基线与渐进优化

## 9. 成功标准

- 用户能更放心地使用排序能力，并可随时回退
- 新用户在首次打开插件时更容易完成配置与理解
- 搜索、定位、聚焦和切换动作更快完成
- 高频动作尽量在树内闭环，减少跳出插件的次数
- 新增能力不破坏当前“轻量导航器”的清晰定位
