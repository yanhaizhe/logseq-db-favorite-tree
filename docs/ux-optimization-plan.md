# UI/UX 优化与功能演进计划

本文档从产品设计与 UI/UX 专家视角，对 `Logseq DB Favorite Tree` 当前实现的风格、交互、可访问性、功能完整度进行系统性审视线，并提出分阶段优化建议。

## 审视线范围

基于对 `src/render.ts`、`src/sidebar-render.ts`、`src/style.css`、`src/floating-layout.ts`、`src/wire-dom-events.ts`、`src/plugin.ts` 等全部源码的通读，结合面板与侧边栏双模式的实际渲染输出，形成以下评估。

---

## 一、当前 UI/UX 优势（值得保留）

| 维度 | 亮点 |
|---|---|
| 视觉层次 | 悬浮面板的三段式布局（header / controls / body / footer）分区清晰，信息密度适中 |
| 微交互 | hover 上浮 (translateY(-1px))、按钮状态切换 (is-active)、刷新动画均已落地 |
| 主题适配 | 全面使用 Logseq CSS 变量，明暗主题自动跟随，无硬编码色值 |
| 空状态引导 | 无收藏 / 无层级 / 搜索无结果 / 加载失败四类空状态均有差异化文案与操作入口 |
| Tooltip 体系 | 悬浮面板与侧边栏共用同一 tooltip 渲染函数，位置策略做了 header actions 区单独适配 |
| 性能摘要 | footer 区展示刷新耗时/渲染耗时/索引构建耗时，对开发调试友好 |
| 刷新动画 | 顶部渐变进度条 + 刷新按钮旋转，视觉反馈到位 |

---

## 二、风格与视觉优化建议

### 2.1 树节点视觉连接线（P0）

**现状**：悬浮面板使用 `margin-left: calc(var(--depth) * 14px)` 纯缩进表示层级；侧边栏使用 `border-left` + 子节点 `margin-left: 18px`。两者均缺乏父-子之间的纵向连线，深层嵌套时视觉追踪困难。

**建议**：

```
└─ 根节点 A              ← 当前：纯缩进，无纵向连线
   ├─ 子节点 B            ← 建议：仿 VS Code / Finder 树形连线
   │  ├─ 孙节点 C
   │  └─ 孙节点 D
   └─ 子节点 E
```

- 使用 `::before` 伪元素绘制纵向连线（类似侧边栏已有的 `border-left` 方案但延伸到每一层）
- 最后一层子节点的连线末端加圆角收口
- 连线颜色使用 `var(--ls-border-color)` 并降低透明度，避免喧宾夺主
- 悬停时连线颜色略微加深，与该行 hover 背景形成呼应

**实现路径**：纯 CSS，在 `style.css` 中为 `.tree-node__children` 补充连线伪元素即可，不涉及 JS 逻辑变更。

### 2.2 侧边栏模式视觉收敛（P1）

**现状**：侧边栏模式的控件区（搜索框 + 工具栏按钮）与 Logseq 原生侧边栏风格存在微妙差异——原生侧边栏更紧凑、按钮更小、间距更窄。

**建议**：
- 侧边栏模式工具栏按钮高度从 24px 降至 20px，字号从 12px 降至 11px
- 搜索框高度从 28px 降至 24px，与 Logseq 原生搜索框高度对齐
- 控件区 padding 从 `0 12px 8px` 收紧为 `0 8px 4px`
- 工具栏按钮间距从 6px 收紧为 4px
- Heading 字号与 Logseq 原生 "Favorites" 标题对齐

### 2.3 悬浮面板工具栏紧凑模式（P1）

**现状**：工具栏按钮始终为图标 + 文字组合，搜索模式下按钮行数过多时折行，占用较多垂直空间。

**建议**：
- 增加「紧凑模式」切换：工具栏从 icon+text 变为 icon-only（26px × 26px 图标按钮）
- 紧凑模式下按钮排列更密集，适合小屏或偏好精简界面的用户
- 紧凑模式状态跟随 graph 持久化

### 2.4 当前页路径高亮面包屑（P2）

**现状**：面包屑仅在面板体内以文字形式展示，路径较深时占用一行空间，且与树节点的定位标记（badge + 高亮）重复传达"当前页位置"信息。

**建议**：
- 面包屑可选折叠为一行，超出宽度时省略号截断，hover 展开完整路径
- 当前页在树中的行已用 `is-current` 高亮 + badge 标记，面包屑可作为辅助信息而非必显示元素
- 考虑将面包屑合并到 header 区域，或作为可选的紧凑内联元素

### 2.5 滚动条样式统一（P2）

**现状**：侧边栏模式的 `__body` 区没有自定义滚动条样式（依赖浏览器默认），而悬浮面板已有完整的 `::-webkit-scrollbar` 定制。

**建议**：为侧边栏 `__body` 补充与悬浮面板一致的轻量滚动条样式（`scrollbar-width: thin` + 自定义 thumb/track）。

## 三、交互优化建议

### 3.1 树内键盘导航（P0）

**现状**：插件的键盘支持仅限于搜索框输入、新建子页面输入框的 Enter/Esc、悬浮球展开的 Enter/Space。树节点本身完全依赖鼠标点击，无法用键盘浏览。

**建议**：实现标准树形组件键盘导航：

| 按键 | 行为 |
|---|---|
| `↓` / `↑` | 在同级或可见节点间上下移动焦点 |
| `→` | 展开当前焦点节点（若已折叠） |
| `←` | 折叠当前焦点节点（若已展开）；若已折叠则跳转到父节点 |
| `Enter` | 打开当前焦点页面 |
| `Space` | 切换展开/折叠 |
| `Tab` | 焦点在搜索框 → 树区域 → 工具栏之间循环 |

**要求**：
- 焦点样式使用 `:focus-visible` 的 outline ring，与 Logseq 原生焦点风格一致
- 键盘导航时自动滚动到焦点节点
- 侧边栏与悬浮面板两套 UI 共用同一键盘导航逻辑

### 3.2 右键上下文菜单（P0）

**现状**：节点行内已有「新建子页面」和「右侧栏打开」两个内联按钮，但缺少更多操作入口，且两个按钮在窄面板或长标题时会被挤压。

**建议**：为树节点行增加右键（或长按）上下文菜单，包含：

- 打开页面
- 在右侧栏打开
- 新建子页面
- 复制页面引用 (`[[页面名]]`)
- 复制页面链接
- 展开全部子节点 / 折叠全部子节点
- 排序模式切换 / 清除自定义排序（仅当有自定义排序时显示）
- （未来）加入收藏 / 移出收藏

**实现路径**：
- 使用 `<menu>` 原生弹出或自定义定位弹出层
- 菜单出现位置基于鼠标/触摸点，做视口边界检测
- 菜单项过多时分组合并，加分隔线

### 3.3 拖拽排序落点增强（P0）

**现状**：拖拽时落点标记仅为 `is-sort-before` / `is-sort-after` 两个 CSS 类，在行顶部/底部产生 2px 的 `box-shadow` 内阴影。视觉反馈较弱，尤其在深色主题下不易察觉。

**建议**：
- 落点指示改为行间的显式插入线（4px 高亮横线，`var(--ft-accent)` 色），而非内阴影
- 拖拽源行加上半透明效果（`opacity: 0.5`）以区分
- 拖拽时显示一个跟随鼠标的半透明缩略图（使用 `setDragImage`）
- 拖拽成功后短暂闪烁目标位置

### 3.4 搜索体验增强（P1）

**现状**：搜索基于精确子串匹配（`key.includes(normalizedQuery)`），不支持模糊匹配、拼音或 typo 容错。

**建议**：
- 增加简单的模糊匹配策略：对标题的连续字符匹配打分，不要求完整子串
- 搜索结果面板可考虑折叠为只显示第一层命中 + "展开 N 个更多结果" 的折叠模式
- 搜索框增加清除按钮（× 图标），一键清空搜索词
- 搜索历史记录（最近 5 次搜索），点击即可复用
- 无结果时给出更具体的建议（"试试搜索英文标题" / "检查拼写" / "刷新索引"）

### 3.5 悬浮球信息密度提升（P1）

**现状**：悬浮球仅显示文件夹图标 + 根节点计数，信息密度很低。用户收回为悬浮球后失去所有上下文。

**建议**：
- 悬浮球增加 tooltip 显示最近一次刷新时间和根节点数
- 可选显示当前打开页面名称（若在树中）
- 当有新 DB 变更但未刷新时，悬浮球显示小圆点通知标记

### 3.6 批量操作支持（P2）

**现状**：所有操作均为单节点操作，无法批量展开/折叠、批量移动排序。

**建议**：
- `Shift + 点击` 范围选择同级节点
- `Ctrl/Cmd + 点击` 多选不连续节点
- 选中后提供批量折叠/展开操作入口

## 四、功能补充建议

### 4.1 收藏管理入口（P1）

**现状**：插件完全依赖 Logseq 原生收藏夹界面来添加/移除收藏，用户需要切换上下文。

**建议**：
- 树节点右键菜单中增加「加入收藏」/「移出收藏」
- 通过 `logseq.App.getFavorites()` / `logseq.App.setFavorites()` 或等效 API 操作
- 操作后自动刷新树，无需手动刷新
- 若 API 不可用，退化为引导用户到 Logseq 原生界面的提示

### 4.2 节点预览卡片（P2）

**现状**：hover 树节点时只有 tooltip 显示页面名称，无内容预览。

**建议**：
- hover 节点 600ms 后弹出预览卡片，显示页面首段文本/前几条 block
- 卡片使用 `logseq.Editor.getPageBlocks` 获取内容
- 预览卡片靠近节点位置弹出，包含"打开页面"和"在右侧栏打开"快捷按钮

### 4.3 页面移动（跨父节点重挂）（P2）

**现状**：拖拽仅在同级排序，不支持将一个节点拖到另一个父节点下。

**建议**：
- 拖拽时若目标为其他父节点的子节点区域，触发"移动页面"操作
- 实现方式：修改被拖拽页面的 `parent` 属性值
- 需要二次确认弹窗（"将 X 从父节点 A 移至父节点 B？"）
- 这是一个高风险的破坏性操作，需要谨慎实现

### 4.4 展开状态导出/导入（P2）

**现状**：展开状态仅按 graph 存储在 `logseq.settings` 中，无法跨设备同步或备份。

**建议**：
- 提供「导出收藏树配置」功能：将排序、展开状态、显示模式导出为 JSON
- 提供「导入配置」功能：从 JSON 恢复
- 导出文件可存入 Logseq 页面作为备份

## 五、性能优化建议

### 5.1 虚拟滚动（P2）

**现状**：树节点全部渲染到 DOM，收藏根数量多（>50）或深层展开（>200 节点）时首次渲染和刷新可能有明显延迟。

**建议**：
- 对可见树区域实现虚拟滚动（仅渲染视口内 ± 缓冲区的节点）
- 可使用 `IntersectionObserver` 或纯 JS 计算滚动位置
- 侧边栏和悬浮面板两套 UI 可共用同一虚拟滚动逻辑

### 5.2 渲染去抖（P1）

**现状**：搜索输入每个字符都触发完整渲染（`render()`），包括侧边栏 `provideUI` 模板替换。

**建议**：
- 搜索输入增加 150ms debounce，期间仅更新输入框视觉状态而不触发全量渲染
- 本地过滤（如果子节点索引已建立）可以同步执行而不走异步 `ensureChildIndex`

### 5.3 子节点索引缓存粒度（P2）

**现状**：`treeService.invalidateIndex()` 在每次刷新时全量失效，即使是增量 DB 变更。

**建议**：
- 探索增量索引更新：在 `DB.onChanged` 回调中获取变更的 block/page，仅重建受影响子树的索引
- 或者在刷新时保留旧索引作为 fallback，加载新索引成功后再替换

## 六、可访问性（A11y）补充

### 6.1 ARIA 角色与属性（P1）

**现状**：树结构使用 `<div>` 嵌套，缺少 ARIA tree/treeitem 语义标记。

**建议**：
- 树容器添加 `role="tree"` + `aria-label`
- 每个树节点行添加 `role="treeitem"` + `aria-expanded` + `aria-level` + `aria-selected`
- 展开/折叠按钮添加 `aria-label`
- 子节点容器添加 `role="group"`

### 6.2 焦点管理（P1）

**现状**：面板打开/关闭、模式切换时焦点可能丢失或停留在已移除的元素上。

**建议**：
- 面板打开时自动聚焦搜索框（除非用户正在编辑页面内容）
- 面板关闭时归还焦点到触发元素（工具栏按钮）
- 模式切换时保持搜索框焦点
